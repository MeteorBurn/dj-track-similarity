# Анализируйте библиотеку из интерфейса

> Для кого: Для тех, кто запускает задачи анализа в браузере.
> Задача: Объяснить выбор моделей, прогресс задач и поведение сброса.
> Тип: Руководство

Переходите к анализу после сканирования. Сканирование находит названия и теги, но не отвечает на
вопросы «что звучит близко к этому треку?» или «что подходит выбранному движению энергии?». Анализ
создаёт локальные данные, необходимые для таких ответов.

Обычно трек достаточно проанализировать один раз для каждого контракта модели, а затем использовать
сохранённый результат во многих поисках. Анализ читает аудио и записывает SQLite, но не изменяет
исходные аудиофайлы.

## Выберите полезный первый результат

| Первый результат | Что выбрать |
| --- | --- |
| Найти треки рядом со знакомым ориентиром | MERT |
| Искать с управлением по слышимым признакам | SONARA |
| Искать по текстовому описанию звука | CLAP |
| Создавать полноценные черновики SET и Hybrid | SONARA, MAEST, MERT и CLAP |
| Сравнивать отдельные модельные окружения в LAB | Добавить MuQ |
| Применять признак, обученный в Rhythm Lab | CLASSIFIERS после появления необходимых входов |

Начните с малого лимита и знакомой музыки. Сначала выясните, какие результаты полезны именно для
вашей коллекции, и только потом запускайте долгий анализ всей библиотеки.

## Выберите модели

В интерфейсе доступны:

- **SONARA** позволяет точнее сформулировать вопрос о ритме, звучании, динамике, гармонии и темпе.
  Она также даёт SET сведения об энергии и переходах.
- **MAEST** добавляет жанроподобные метки и ещё один взгляд на аудиосходство для SET и Hybrid.
- **MERT** создаёт широкое аудиоокружение для вопроса «что находится рядом с этим треком?».
- **MuQ** добавляет отдельную колонку сравнения в LAB Reference Compare.
- **CLAP** связывает текстовые описания со звуком и предоставляет SET и Hybrid ещё один сигнал
  аудиосравнения.
- **CLASSIFIERS** применяет опубликованные личные профили после появления входов, объявленных каждым
  профилем.

В компактном блоке анализа есть одна кнопка **Analyze**. При открытии страницы выбраны SONARA и
Core. SONARA, группа ML-моделей и CLASSIFIERS — отдельные семейства: выбор одного снимает флажки
других семейств, но внутри ML можно оставить несколько моделей. **FULL** явно выбирает все этапы.
Кнопка запускает ровно отмеченные модели, а после старта флажки не меняются. Все этапы используют
одну последовательную очередь в памяти.

## Технические подробности результатов SONARA

SONARA выполняется отдельно от ML-моделей и классификаторов. Задача передаёт партии путей в
`sonara.analyze_batch()`, где декодированием управляет Symphonia. В задаче SONARA нет FFmpeg проекта,
`DecodedAudio`, `analyze_signal` или резервной обработки по одному файлу. Для BPM проект задаёт
диапазон `70.0..180.0`.

Сценарии, зависящие от темпа, начинают с актуальных подписанных данных SONARA, но не доверяют им
безусловно. При уверенности ниже `0.45` преобразователь проверяет кандидатов SONARA и тег BPM из
Mutagen. Подтверждённый тег может стать рабочим BPM. `grid_stability` способен снизить надёжность и
сдвинуть темповую оценку к нейтральной. Если треки анализировались до настройки текущего диапазона,
снова выберите SONARA и запустите анализ. Старая сигнатура не совпадёт, поэтому треки автоматически
попадут в очередь без сброса.

Результат Core по умолчанию для SONARA v0.2.9 также хранит исходный BPM, `bpm_confidence`,
кандидатов темпа, тональность Camelot, компактные данные структуры и громкости, vocalness v2,
настроение и другие небольшие значения. Диалог метаданных показывает их рядом с происхождением
анализа: версией схемы, частотой дискретизации, длиной шага, режимом анализа, запрошенными признаками
и установленной версией пакета.

## Выберите результаты SONARA

При выборе SONARA появляются три флажка:

- **Core** (по умолчанию) записывает небольшие скалярные значения и векторы фиксированного размера в выбранную основную базу.
- **Timeline** записывает полные временные массивы, события и сегменты в соседнюю базу `*.timeline.sqlite`.
- **Representations** записывает эмбеддинг и отпечаток SONARA в соседнюю базу `*.representations.sqlite`.

Можно сначала запустить Core, а Timeline или Representations добавить позже. Одна нативная партия
запрашивает у Rust объединение выбранных результатов. У каждого типа собственная детерминированная
сигнатура, поэтому добавление Timeline не делает актуальный Core устаревшим.

Mood и instrumentalness сохраняются и отображаются, но сейчас не участвуют в сходстве, SET, Hybrid
или классификаторах. True peak и ReplayGain хранятся для возможного будущего управления громкостью,
а не прямой оценки сходства SONARA. Полные позиции битов и атак, метки и события аккордов, кривые
темпа, энергии и громкости, массивы сильных долей и структурные сегменты относятся к Timeline.
Эмбеддинг и отпечаток SONARA относятся к Representations.

Диалог метаданных показывает все значения Core. Для Timeline и Representations он выводит только
`Data present` и точные имена сохранённых полей, не загружая тяжёлые данные ради открытия окна.
Timeline API остаётся доступным для будущих сценариев, которым действительно нужен массив.

## Поведение лимита

В интерфейсе `Analyze limit = 0` означает всю библиотеку. Положительный лимит считает треки, у
которых нет выбранных результатов.

В CLI правило другое: для всей библиотеки параметр `--limit` нужно опустить.

## Вычислительное устройство

- `AUTO` выбирает CUDA, если PyTorch видит GPU, иначе CPU.
- `CPU` принудительно использует процессор.
- `CUDA` запрашивает CUDA и должен завершиться с понятной ошибкой, если CUDA недоступна.

SONARA работает на CPU. MAEST, MERT, MuQ и CLAP используют выбранное устройство через свои адаптеры.
MuQ загружает официальные веса `OpenMuQ/MuQ-large-msd-iter` и получает только аудио `float32` с
частотой 24 кГц. Сейчас MuQ поддерживает LAB Reference Compare и не входит в SET, Hybrid, Audio Dedup
или расчёт классификаторов.

## Управление партиями

- **SONARA native batch**: `1..16` путей, по умолчанию `8`. Отмена вступает в силу между возвращёнными партиями.
- **Track batch size**: `1..64` декодированных треков, удерживаемых и обрабатываемых вместе. Измеренное значение по умолчанию — `8`.
- **Inference batch size**: `1..128` примеров MAEST/MERT/MuQ/CLAP за один проход модели. Измеренное значение по умолчанию для RTX 3090 — `16`.

Значение SONARA определяет число одновременных нативных чтений полных файлов, а не размер инференса.
Для библиотеки на одном HDD оставьте настройку по умолчанию. Повышайте её только после измеренного
пилотного запуска, который показывает, что накопитель справляется с дополнительными параллельными
чтениями. В задачах только SONARA кандидаты обрабатываются по пути, чтобы соседние партии сохраняли
локальность файловой системы, а не прыгали между папками, отсортированными по исполнителю.

## Прогресс и журналы

Интерфейс периодически запрашивает текущую задачу и показывает:

- состояние: queued, running, completed, cancelled или failed;
- общее, обработанное, проанализированное, ошибочное и пропущенное количество;
- текущую модель и путь;
- прогресс каждой модели;
- журнал событий и ошибки.

Сообщение очереди также отражает только активный этап. Для SONARA показаны выбранные результаты и
партия SONARA. Для ML — модели, Device, Track batch и Inference batch. Для CLASSIFIERS — число
выбранных профилей без настроек ML.

Первичная загрузка базы и треков не зависит от расчёта готовности классификаторов, поэтому проверка
артефактов не задерживает основную таблицу библиотеки.

После каждой завершённой партии SONARA журнал отдельно показывает длительности `analyze`, `prepare`
и `store`, а также скорость нативного чтения в МиБ/с. Выбранные результаты Core, Timeline и
Representations фиксируются одной транзакцией SQLite на партию. Точка сохранения для каждого трека
не позволяет ошибке дополнительной базы оставить трек частично обновлённым и при этом даёт остальным
трекам партии завершиться.

Квадратная кнопка остановки запрашивает отмену, но не прерывает Python посреди записи. Партия SONARA
сохраняется только после возврата нативного вызова, а отмена проверяется перед следующим фрагментом.

## Конвейер и готовность классификаторов

Конвейер **Analyze** всегда выполняет SONARA, затем ML и затем CLASSIFIERS независимо от порядка
выбора. Ошибки отдельных файлов остаются видимыми, но не останавливают последующие этапы. Критическая
ошибка инициализации или отмена останавливает родительскую задачу и отменяет ещё не начавшиеся этапы.

CLASSIFIERS никогда не декодирует аудио. Блок показывает готовые и неготовые количества и причины
блокировки манифестов. Общее число включает пары классификатор–трек, которые готовы, но не имеют
оценки либо имеют оценку другого идентификатора модели. Запустите этап позже, чтобы добавить треки,
ставшие готовыми после расширения покрытия SONARA или ML.

## Кнопки сброса

Сброс изменяет только SQLite:

- сброс SONARA удаляет признаки Core, строки Timeline, эмбеддинг и отпечаток SONARA, флаги и зависимые оценки классификаторов основной библиотеки; метки и обратная связь сохраняются;
- сброс MAEST удаляет метаданные и эмбеддинги MAEST;
- сброс MERT, MuQ или CLAP удаляет эмбеддинги соответствующего ключа;
- сброс CLASSIFIERS удаляет выбранные строки `track_classifier_scores`.

Перед первой нативной задачей SONARA данные старого контракта становятся блокировкой. Создайте
резервную копию базы и используйте явный сброс SONARA: приложение не адаптирует, не смешивает и не
удаляет такие строки автоматически. После этого частичное покрытие актуальными результатами можно
обычно продолжать по сигнатуре.

Полный порядок для существующей библиотеки, включая резервные копии и пересборку классификаторов,
описан в разделе
[Повторный анализ SONARA с раздельным хранением](../workflows/reanalyze-sonara-split-storage.md).
