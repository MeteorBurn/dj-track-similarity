# Повторно проанализируйте SONARA с раздельным хранением

> Для кого: Для тех, кто открывает фонотеку со схемой v5 в текущей версии приложения.
> Задача: Перейти на хранилища Core, Timeline и Representations и заново выполнить анализ без смешения данных.
> Тип: Сценарий

## 1. Создайте резервную копию каталога

Остановите сервер и скопируйте основную базу вместе со всеми существующими соседними базами. После
начала анализа всегда копируйте все три файла вместе: они образуют один каталог.

## 2. Откройте основную базу

Откройте существующий файл `.sqlite` со схемой v5 в текущем приложении. Однократная миграция до
схемы v6 создаст:

```text
library.sqlite
library.timeline.sqlite
library.representations.sqlite
```

Миграция намеренно очищает старые данные SONARA и делает недействительными оценки классификаторов,
зависящих от SONARA. Существующие эмбеддинги MAEST/MERT/MuQ/CLAP, метаданные MAEST, их флаги
анализа, оценки классификаторов только по эмбеддингам, треки, файловые теги, отметки «нравится»,
обратная связь и записи оценивания сохраняются.

Базы старше схемы v5 не адаптируются. Вместо этого просканируйте фонотеку в новую базу текущей
версии.

## 3. Явно сбросьте старый контракт SONARA

Если строки Core, Timeline или Representations созданы по более раннему контракту проекта,
предварительная проверка нативной задачи заблокирует запуск. После резервного копирования используйте
существующий сброс SONARA. Он удалит все три хранилища SONARA и зависимые оценки классификаторов, не
изменяя аудио, метки, обратную связь или ML-эмбеддинги, не зависящие от SONARA. Старые и новые данные SONARA не
адаптируются и не смешиваются.

## 4. Выполните анализ SONARA

Используйте флажки интерфейса или CLI. По умолчанию анализируется только Core:

```powershell
dj-sim analyze --models sonara --db C:\db\library.sqlite
```

Для всех актуальных данных SONARA:

```powershell
dj-sim analyze --models sonara --sonara-outputs core,timeline,representations --db C:\db\library.sqlite
```

Необязательные результаты можно добавить позднее:

```powershell
dj-sim analyze --models sonara --sonara-outputs timeline,representations --db C:\db\library.sqlite
```

Размер нативной партии по умолчанию — `8`. При необходимости используйте
`--sonara-batch-size 1..16`. Завершите полный проход Core до переобучения классификаторов, зависящих
от SONARA. Полные проходы Timeline и Representations нужны только тогда, когда эти результаты
выбраны явно.

## 5. Отдельно добавьте отсутствующие ML-модели

```powershell
dj-sim analyze --models maest,mert,muq,clap --db C:\db\library.sqlite
```

SONARA нельзя объединять с этими моделями или классификаторами в одной задаче. Миграция не требует
повторного запуска моделей, если их эмбеддинги уже существуют.

## 6. Отдельно переобучите, опубликуйте и рассчитайте классификаторы

Каждый классификатор, зависящий от SONARA, переобучайте и публикуйте только после полного покрытия
необходимыми данными Core и ML. Затем запустите:

```powershell
dj-sim analyze-classifiers --db C:\db\library.sqlite
```

Задача оценивает только треки с полным актуальным набором входов для конкретного манифеста. Повторный
запуск подхватит треки, которые станут готовыми позже.

## 7. Проверьте метаданные

Откройте диалог метаданных трека:

- Core должен показывать рассчитанные значения и происхождение SONARA `0.2.9`/schema `4`.
- Timeline должен сообщать о наличии данных и перечислять имена сохранённых полей.
- Representations должен сообщать о наличии данных и перечислять `embedding` и `fingerprint` SONARA.

Проверьте готовность и блокировки классификаторов в блоке CLASSIFIERS, затем убедитесь в ожидаемом
покрытии оценками. После резервного копирования и сброса сначала выполните нативный пилот на 100
файлах. Не переходите к полному Core, пока пилот не завершится без ошибок с нативными подписями, а
проверка целостности SQLite не пройдёт успешно.
