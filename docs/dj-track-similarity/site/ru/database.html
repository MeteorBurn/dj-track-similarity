База данных и сохраняемые данные ​

Эта страница описывает текущую схему SQLite и метаданные/данные анализа, которые сохраняются в SQLite. Используйте её, чтобы понять, что именно сохраняет приложение, что удаляет сброс и почему тот или иной режим поиска сначала требует определённого прохода анализа.

Спецификация SQLite ​

Текущая версия схемы — 3.

Существующие library DB со схемой версии 2 не мигрируются приложением автоматически. Сначала запустите отдельный migration script; dry-run является режимом по умолчанию:

powershell
.\.venv\Scripts\python.exe scripts\migrate_database_v3.py --db .\data\library.sqlite
.\.venv\Scripts\python.exe scripts\migrate_database_v3.py --db .\data\library.sqlite --apply

Перед --apply скрипт создаёт online SQLite backup. Закройте запущенное приложение перед применением миграции к той же базе. Этот же скрипт backfill'ит v3 FTS search table для баз, которые уже были переведены на v3 до появления FTS индекса; это всё равно явное действие скрипта, а не runtime migration.

База данных — это локальное состояние пользователя. Обычные рабочие процессы сканирования, анализа, поиска, сброса, очистки и переноса библиотеки изменяют только записи в SQLite; они не перезаписывают аудиофайлы. Явным исключением является отдельный рабочий процесс записи жанровых тегов MAEST, описанный в разделе Поиск и теги.

tracks ​

Хранит по одной строке на каждый проиндексированный аудиофайл:

  • id: стабильный локальный идентификатор трека.
  • path: уникальный сохранённый путь к аудиофайлу.
  • size: размер файла на момент сканирования.
  • mtime: время изменения файла на момент сканирования.
  • artist, title, album: выбранные метаданные файла.
  • bpm, musical_key, energy, duration: рабочие поля, используемые интерфейсом и процессами анализа.
  • has_sonara_analysis, has_maest_embedding, has_mert_embedding, has_clap_embedding: производные флаги наличия анализа, поддерживаемые analysis writes и resets. Они ускоряют счётчики summary и выбор кандидатов для missing-analysis. Это флаги наличия сохранённых данных, а не политика stale/fresh.
  • metadata_json: объект JSON для полей Mutagen и метаданных, полученных от моделей.
  • created_at, updated_at: локальные временные метки строки.

metadata_json должен быть корректным JSON. В схеме есть триггеры, которые отклоняют некорректный JSON при вставке или обновлении.

Схема v3 содержит partial indexes для missing и present analysis flags. Эти индексы используются счётчиками summary и выбором кандидатов для analyzer, так что анализ почти полной большой библиотеки не должен сканировать всю таблицу tracks, чтобы найти небольшой набор строк без SONARA, MAEST, MERT или CLAP.

Используйте эту таблицу, чтобы ответить на вопросы «какие треки есть в библиотеке?» и «какие метаданные или сводки анализа интерфейс показывает для строки?». Поле path — это ссылка обратно на локальный аудиофайл; перенос библиотеки обновляет этот сохранённый путь только тогда, когда те же файлы были перемещены в новый корневой каталог.

embeddings ​

Хранит векторы моделей по треку и пространству embedding:

  • track_id: ссылается на tracks.id.
  • embedding_key: в настоящее время mert, clap или maest.
  • model_name: идентификатор модели или контрольной точки.
  • dim: размерность вектора.
  • vector: бинарная полезная нагрузка вектора float32.
  • updated_at: локальная временная метка строки.

Первичный ключ — (track_id, embedding_key), поэтому один и тот же трек может иметь векторы MERT, CLAP и MAEST без смешивания этих пространств.

Используйте эту таблицу, чтобы проверить, достаточно ли данных для рабочего процесса на основе embedding: поиску MERT нужен mert, текстовому поиску CLAP нужен clap, а продвинутым combined-классификаторам нужны mert и maest вместе с признаками Sonara.

library_settings ​

Хранит локальные настройки уровня базы данных, например выбранный корневой каталог музыки.

Используйте эту таблицу для предпочтений уровня приложения, привязанных к одной базе данных, а не к отдельному треку.

track_likes ​

Хранит локальный список понравившихся треков:

  • track_id: ссылается на tracks.id.
  • liked_at: локальная временная метка последнего действия «лайк».

Первичный ключ — track_id, поэтому трек либо отмечен лайком, либо нет. Строки удаляются автоматически, когда соответствующий трек удаляется из локальной базы данных библиотеки. Лайки — это только состояние приложения; они никогда не записываются в аудиофайлы или теги Mutagen.

track_classifier_scores ​

Хранит производные выходные данные классификатора по треку и ключу классификатора:

  • track_id: ссылается на tracks.id.
  • classifier: ключ классификатора, например live_instrumentation.
  • score: основная оценка, видимая пользователю и используемая для фильтрации.
  • label: укрупнённая метка, например high, medium или low.
  • confidence: максимальная вероятность класса.
  • probabilities_json: вероятности классификатора по обучающим меткам профиля.
  • feature_set: семейство признаков, используемое артефактом классификатора, в настоящее время combined.
  • model_id: путь к продвинутой модели, использованной для оценки.
  • analyzed_at: локальная временная метка оценки.

Первичный ключ — (track_id, classifier), поэтому повторный запуск классификатора обновляет оценку для этого трека, а не добавляет исторические строки.

Используйте эту таблицу, когда фильтр CLASS ведёт себя не так, как ожидается. Отсутствующие строки обычно означают, что продвинутый классификатор ещё не оценил трек или что у трека не было необходимых входных данных Sonara, MERT или MAEST во время оценки.

track_search_fts ​

Хранит производный FTS5 индекс для явного token-based поиска по библиотеке:

  • track_id: стабильный локальный track ID, сохранённый unindexed для join обратно к tracks.id.
  • search_text: материализованный текст artist, title, album, path и metadata_json.

Обычный поиск библиотеки по умолчанию использует substring LIKE. FTS используется только когда caller явно передаёт search_mode=fts. FTS token matching может сильно ускорить count/filter для широких запросов, но sorted first page всё ещё может упираться в сортировку library order, если token очень частый. Scan/upsert, RefreshTags, MAEST genre saves, Sonara metadata saves, resets, меняющие metadata, library relocation и clear-library поддерживают FTS rows в той же SQLite write transaction, что и связанное изменение tracks.

Метаданные и данные анализа ​

Приложение намеренно разделяет теги файлов и вычисляемые значения.

Сканирование Mutagen читает следующий фиксированный список (whitelist):

  • artist
  • title
  • album
  • genre
  • year
  • country
  • label
  • catalog_number
  • track_number
  • disc_number
  • bpm
  • key
  • comment
  • isrc
  • duration
  • audio_format
  • audio_codec
  • date

Значения нормализуются в JSON-безопасные значения перед сохранением. Специфичные для Mutagen объекты, такие как временные метки ID3, преобразуются в строки.

RefreshTags заменяет только это подмножество метаданных Mutagen. Он сохраняет сохранённые пути и данные анализа моделей.

Выходные данные анализа находятся рядом с тегами файлов, а не заменяют их. Именно поэтому трек может одновременно показывать метаданные файла, такие как genre или bpm, и вычисляемые значения, такие как BPM от Sonara, жанры MAEST, embeddings или оценки классификаторов.
