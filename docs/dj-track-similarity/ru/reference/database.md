# Справочник базы данных

> Для кого: Пользователи и сопровождающие, которым нужно понимать локальное состояние SQLite.
> Задача: Объяснить базу как локальное состояние без полного дампа схемы.
> Тип: Справочник

При выборе `library.sqlite` приложение открывает единый каталог из трёх соседних файлов:

| Хранилище | Файл | Содержимое |
| --- | --- | --- |
| Core | `library.sqlite` | треки, теги, лёгкие значения SONARA, эмбеддинги MAEST/MERT/MuQ/CLAP, флаги, отметки, состояние классификаторов и оценок, FTS |
| Timeline | `library.timeline.sqlite` | полные массивы SONARA, временные события, кривые и сегменты |
| Representations | `library.representations.sqlite` | только необязательные представления SONARA: эмбеддинг и отпечаток |

Во всех трёх файлах хранится один общий созданный идентификатор каталога. Приложение отклоняет дополнительную базу,
скопированную из другого каталога. При переносе и резервном копировании держите их вместе.

## Состояние основной библиотеки

База хранит:

- пути треков, размер, время изменения и выбранные метаданные;
- рабочие BPM, тональность, энергию, длительность, исполнителя, название и альбом;
- флаги наличия анализа SONARA, MAEST, MERT, MuQ и CLAP;
- одну индексированную таблицу Core `embeddings` с векторами MAEST, MERT, MuQ и CLAP в формате float32 для поиска и ранжирования;
- метаданные SONARA Core: `bpm_confidence`, настроение, instrumentalness, громкость, компактную структуру, сетку долей, варианты тональности, vocalness, тишину, Contrast, MFCC и Chroma;
- `sonara_provenance` с настройками анализа SONARA и установленной версией пакета, если она доступна;
- `sonara_analysis_signature` — детерминированный контракт совместимости SONARA, схемы, режима, частоты, диапазона BPM, профиля запроса и ревизии проекта;
- списки имён полей Timeline и Representations для диалога метаданных;
- понравившиеся треки;
- оценки по `classifier_key`;
- строки FTS для поиска библиотеки;
- настройки библиотеки, включая данные опубликованного профиля оценки, если они используются.

База Timeline сохраняет все возвращённые последовательности. Representations хранит только
необязательный 48-мерный вектор SONARA как BLOB в формате float32 и отпечаток как JSON. Обычное чтение трека
возвращает имена полей, а не значения. Показатели настроения и instrumentalness пока только
сохраняются. True peak и ReplayGain оставлены для возможного управления громкостью; прямой расчёт похожести
SONARA их игнорирует.

`GET /api/tracks/{track_id}/sonara-timeline` явно загружает данные Timeline для будущего сценария. Диалог
метаданных эту конечную точку не вызывает.

Физический флаг `has_sonara_analysis` описывает только Core. Актуальность Timeline и Representations
определяется их собственными строками и сигнатурами. Планирование проверяет точные выбранные
результаты.

## Миграция схемы

При открытии основной базы схемы v5 она мигрирует до v6 и создаёт две дополнительные базы. При этой
миграции анализ SONARA намеренно начинается заново. Каталог и метаданные MAEST сохраняются вместе с эмбеддингами
MAEST/MERT/MuQ/CLAP и их флагами. Также остаются отметки «нравится», обратная связь, строки оценки и
оценки классификаторов только по эмбеддингам. Старые значения и кривые SONARA удаляются, сбрасывается
только флаг Core, а зависимые оценки классификаторов становятся недействительными. Базы схемы v4 и
старше отклоняются, а не адаптируются.

Ревизия признаков классификаторов SONARA также хранится в `library_settings`. При первом открытии
основной базы после изменения ревизии оценки становятся недействительными, если `feature_set`
равен `combined` или любой разделённый плюсом источник начинается с `sonara`. Это включает
`sonara`, `sonara2`, `sonara2vocal` и их сочетания с эмбеддингами. Rhythm Lab независимо хранит ту же
ревизию и удаляет зависимые предсказания при открытии базы меток. Оценки и предсказания только по
эмбеддингам, метки Rhythm Lab, отметки «нравится», обратная связь по парам и переходам сохраняются.
Старые файлы моделей остаются для восстановления, но манифесты SONARA без актуальной сигнатуры нельзя
использовать для расчёта оценок. Требуются переобучение и новая публикация.

Не путайте версии: схема результата SONARA — `4`, основная схема SQLite — `6`, ревизия признаков
SONARA проекта — `5`, а версия манифеста опубликованного классификатора — `2`. Ревизия `5` также
подписывает `decoder_backend="sonara-symphonia"` и `execution_path="analyze_batch"`.

## Оценка и обратная связь

Поиск и диагностика Hybrid могут записывать локальные строки оценки: сессии, результаты, обратную
связь и калибровку.

## Состояние Rhythm Lab

По умолчанию Rhythm Lab использует отдельную базу меток под `tools/rhythm-lab/data/`. В ней находятся
профили, метки, предсказания, очереди, коллекции, контрольные точки и ревизия признаков, использованных для предсказаний
SONARA. При смене ревизии удаляются только зависимые предсказания. Метки и обратная связь остаются
для переобучения.

## Границы записи

- Scan, Refresh Tags, анализ, сброс, очистка, отметка «нравится», расчёт классификаторов, обратная связь и применение переноса записывают данные в SQLite.
- Применение переноса меняет только сохранённые пути.
- Очистка базы удаляет треки Core и связанные строки Timeline/Representations, затем пересоздаёт FTS. Аудиофайлы не удаляются.
- Сброс SONARA удаляет метаданные Core, строки Timeline, эмбеддинг и отпечаток SONARA и зависимые оценки классификаторов.
- Сброс SONARA удаляет сведения о происхождении и сигнатуру вместе с метаданными и флагом наличия, но сохраняет метки и обратную связь.
- Применение Audio Dedup удаляет строки SQLite только для файлов, которые удалось удалить.

## Резервное копирование

Перед разрушительным обслуживанием реальной базы сохраните все три соответствующих файла или
работайте с копией. Поддерживаемый скрипт обслуживания делает это автоматически.
