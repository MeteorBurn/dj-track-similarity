Модели анализа ​

Эта страница описывает модели, лежащие в основе каждого семейства анализа: что представляет собой каждая модель, что она выдаёт и как приложение использует и сохраняет её результат. Используйте её вместе с Анализ, где основное внимание уделено тому, какой проход запускать первым, и Поиск и теги, где описано, как выполняется поиск по результатам.

Весь анализ моделей работает по принципу database-first: он читает аудио и записывает результаты только в SQLite. Он не изменяет аудиофайлы. Единственный явный путь записи в аудиофайлы — это сохранение жанровых тегов MAEST, описанное в Поиск и теги.

Признаки Sonara хранятся в metadata_json.sonara_features. Векторы MERT, MAEST и CLAP хранятся в таблице embeddings под своим embedding key. Полную схему хранения см. в разделе База данных.

Выбор устройства для MAEST, MERT и CLAP подчиняется одному правилу: auto выбирает CUDA, когда PyTorch видит GPU, иначе CPU; явный cuda завершается ошибкой, если CUDA недоступна. Эти три семейства также используют батчинг инференса через --inference-batch-size; батчинг декодированных треков управляется отдельно через --track-batch-size. Группы установки ml и sonara описаны в разделе Установка.

Sonara ​

Sonara — это быстрый объяснимый проход извлечения признаков. Общий загрузчик аудио начинает с sonara.analyze_file, а приложение сохраняет сфокусированный набор признаков плейлиста в metadata_json.sonara_features плюс имя модели в metadata_json.sonara_model.

Она выдаёт сгруппированные, понятные человеку признаки: основные сводки по ритму/громкости/спектру, перцептивные значения (energy, danceability, valence, acousticness), музыкальную тональность с уверенностью и тональный анализ. BPM и тональность Sonara — это проанализированные значения, а не скопированные теги файла, и приложение хранит исходные данные тональности Sonara, не выводя нотацию Camelot.

Как приложение это использует:

  • Вкладка поиска SONARA ранжирует треки по этим признакам с настраиваемыми весами микшера и модификаторами.
  • Поля уровня библиотеки, такие как проанализированные BPM, тональность, energy, danceability и громкость, берутся из Sonara.
  • Признаки Sonara — один из входов для оценки продвинутыми классификаторами.

Sonara работает на CPU и участвует в общих батчах декодированных треков, которые задаются track_batch_size; она не использует neural inference batch. Запустите Sonara первой, если не уверены, с чего начать; это самое прозрачное семейство, потому что UI может напрямую показывать и микшировать её группы признаков. Полный список ключей см. в разделе Анализ.

MAEST ​

MAEST предсказывает жанровые метки и сохраняет embedding MAEST. Адаптер использует maest-infer с моделью discogs-maest-30s-pw-129e-519l.

Она анализирует до трёх 30-секундных окон на трек (около 60-секундного смещения, около 38% длительности и около 72% длительности). Невозможные или дублирующиеся окна ограничиваются и дедуплицируются. Активации по меткам усредняются по окнам для выбора верхних меток, а строки embedding MAEST усредняются по тем же окнам.

Она записывает только в SQLite:

  • metadata_json.maest_model
  • metadata_json.maest_genres
  • metadata_json.maest_syncopated_rhythm
  • embedding под embedding key maest

Как приложение это использует:

  • Сгенерированные жанровые метки поступают в диалог метаданных и необязательный рабочий процесс записи жанровых тегов.
  • maest_syncopated_rhythm управляет библиотечным preset syncopated.
  • Embedding MAEST — вход для оценки продвинутыми классификаторами.

Сам анализ MAEST никогда не изменяет аудио. Сохранение меток в стандартные жанровые теги — это отдельное явное действие, описанное в Поиск и теги.

MERT ​

MERT строит audio-to-audio embeddings под embedding key mert. Модель по умолчанию:

text
m-a-p/MERT-v1-95M

Как приложение это использует:

  • Вкладка поиска MERT находит треки, близкие к выбранным seed-трекам в пространстве embedding MERT.
  • Поиск MERT использует только векторы MERT; он не смешивает признаки Sonara или векторы CLAP.
  • Embeddings MERT — вход для оценки продвинутыми классификаторами.

Результаты зависят от существующих embeddings MERT, поэтому только что просканированные треки должны быть проанализированы до того, как они появятся в полезных результатах MERT. Запускайте MERT, когда схожесть по seed-трекам важнее объяснимых элементов управления.

CLAP ​

CLAP строит ориентированные на музыку аудио-embeddings под embedding key clap и также может встраивать текст в то же пространство для text-to-audio search. Активный чекпойнт:

text
lukewys/laion_clap/music_audioset_epoch_15_esc_90.14.pt

Как приложение это использует:

  • Вкладка текстового поиска CLAP встраивает текстовый prompt и ранжирует относительно него аудиовекторы CLAP.
  • CLI-команда dj-sim text-search делает то же самое из терминала.
  • Текстовый поиск требует аудио-embeddings CLAP, созданных тем же чекпойнтом.

Запускайте CLAP, когда хотите искать по настроению, инструментовке, энергии или другому описательному языку. Чёткие, конкретные prompts обычно работают лучше, чем одно жанровое слово.

Продвинутые классификаторы ​

Продвинутые классификаторы — это не модели анализа аудио; это локальные профили, обученные в Rhythm Lab, которые оценивают треки по уже сохранённым данным SONARA, MERT и MAEST. Поскольку они читают существующий анализ, а не декодируют аудио, сначала запустите Sonara, MERT и MAEST для треков, которые хотите оценить. Подробности об оценке см. в разделе Анализ, а об обучении и продвижении — в Rhythm Lab.
