Идеи ​

Эта страница хранит крупные идеи для будущих рабочих процессов, которые еще не реализованы. Ее нужно воспринимать как черновик продуктового направления: здесь описано, что можно попробовать сделать позже, но это не обещание текущего поведения, не стабильный API-контракт и не список уже готовых функций.

Пояснение: эта страница полезна как место для "а что дальше?". Сюда можно складывать идеи, которые больше одного маленького улучшения UI, но еще не созрели до конкретного технического плана.

Smart Set Builder по seed-трекам ​

Идея: добавить отдельный UI-режим, который генерирует вариативный список кандидатов для DJ-сета на основе 3-10 выбранных seed-треков.

Seed-треки здесь означают несколько треков, которые пользователь вручную выбирает как отправную точку: например, "вот такой грув", "вот такая темнота", "вот такой break", "вот такая абстрактная фактура". Система должна не просто найти еще больше похожих треков, а выступить как куратор поверх уже накопленной аналитики библиотеки.

Цель такого режима:

  • помочь быстро собрать кандидатов для прослушивания;
  • ускорить digging по большой локальной библиотеке;
  • предложить не только очевидные похожие треки, но и полезные соседние варианты;
  • подготовить последовательность, которую можно потом вручную проверить, отредактировать и экспортировать.

Пояснение: обычный similarity search отвечает на вопрос "что похоже на этот трек?". Smart Set Builder должен отвечать на более DJ-ориентированный вопрос: "какой набор треков вокруг этих seed-треков может стать полезной подборкой или началом сета?".

Первая версия без новой модели ​

Первую версию можно сделать без обучения отдельной новой модели. Вместо этого она может комбинировать уже существующие сигналы:

  • MERT similarity для музыкальной и аудиальной близости. Это основной сигнал "похоже по звучанию", особенно когда жанровые теги неполные или шумные.
  • MAEST labels для жанровых и текстурных подсказок. Это помогает понять, что трек относится к похожей музыкальной зоне, даже если MERT-сходство не идеально.
  • SONARA features для BPM, энергии и ритмической совместимости. Это важно именно для DJ-переходов: похожий по настроению трек может быть неудобен, если он слишком далеко по темпу или энергии.
  • promoted classifier scores, например abstract_edge, break_energy или voice_presence, как сигналы предпочтения и исключения. Такие классификаторы отражают личные пользовательские понятия, которые трудно выразить жанром.
  • исключение уже размеченных, лайкнутых, просмотренных или недавно показанных треков, когда пользователь хочет discovery, а не повторную проверку уже известных находок.

Пояснение: это не один "магический" score. Скорее это агрегатор, который берет разные виды уже сохраненной информации и пытается собрать список, полезный для реальной музыкальной работы.

Возможные режимы ​

similar_crate ​

Режим близкой подборки. Он должен держаться рядом с seed-треками и ранжировать кандидатов по сильному совпадению сразу нескольких источников: MERT, MAEST, SONARA и promoted classifier scores.

Пояснение: это режим "дай мне ящик похожего материала". Он нужен, когда seed уже точно попал в нужную зону, и задача не экспериментировать, а быстро найти еще треки рядом.

weird_adjacent ​

Режим странных, но релевантных соседей. Он должен сохранять достаточную похожесть на seed-треки, но предпочитать кандидатов с необычным расхождением моделей или более экспериментальной текстурой.

Пояснение: например, MERT говорит "похоже", а MAEST или классификатор дают неожиданный сигнал. Такой трек может оказаться не очевидным клоном seed-трека, а интересным боковым ходом.

balanced_set ​

DJ-ориентированный режим. Он должен генерировать список с bridge-треками, совместимостью BPM/энергии и меньшим повторением слишком похожих треков подряд.

Пояснение: здесь важен не только выбор отдельных хороших кандидатов, но и порядок. Два трека могут быть отличными по отдельности, но плохо работать рядом из-за энергии, темпа или слишком похожей фактуры.

discovery ​

Режим поиска новых гипотез. Он может допускать более рискованных кандидатов и резервировать часть списка под low-confidence треки, которые потенциально помогут найти новые хорошие positives для пользовательских классификаторов.

Пояснение: это режим "покажи мне то, в чем система не уверена, но что может оказаться полезным". Он особенно связан с active learning: пользователь слушает предложенные треки, отмечает удачные и тем самым улучшает будущие классификаторы.

Полезные контролы в UI ​

В первом UI можно дать пользователю такие настройки:

  • выбор seed-треков;
  • целевая длина списка;
  • уровень diversity, то есть насколько сильно список должен уходить от прямой похожести;
  • energy curve: warmup, balanced, peak или wave;
  • classifier targets и avoid rules;
  • исключение лайкнутых, размеченных, просмотренных или уже экспортированных треков;
  • опциональные ограничения по BPM/key compatibility.

Пояснение по важным контролам:

  • diversity нужен, чтобы управлять балансом между "все очень похоже" и "давай больше неожиданных вариантов".
  • energy_curve нужна, чтобы список был похож не просто на выдачу поиска, а на возможную музыкальную траекторию.
  • classifier_targets позволяют сказать "хочу больше такого свойства", если оно уже выражено promoted classifier.
  • avoid rules позволяют исключить нежелательные свойства, например слишком много вокала или слишком прямой грув, если для этого есть classifier score.

Объяснимый результат ​

Сгенерированный результат должен объяснять, почему каждый трек попал в список. Например:

  • similar_to_seed: трек близок к одному или нескольким seed-трекам.
  • bridge: трек может связывать разные seed-зоны или соседние части списка.
  • weird_adjacent: трек не очевидно похож, но находится рядом по важным признакам.
  • classifier_match: трек выбран из-за сильного совпадения с promoted classifier.

Также результат должен показывать базовые MERT/MAEST/SONARA/classifier scores для ручной проверки.

Пояснение: это важно, потому что пользователь не должен слепо доверять алгоритму. Если система показывает причины выбора и базовые scores, можно быстро понять, почему трек оказался в списке, оставить его, удалить или отправить на дальнейшую разметку.

Возможная форма API ​

Первый API может выглядеть так:

text
POST /api/set-builder/generate

Поля входа:

  • seed_track_ids: список ID выбранных seed-треков.
  • limit: сколько кандидатов вернуть.
  • mode: режим генерации, например similar_crate, weird_adjacent, balanced_set или discovery.
  • diversity: насколько сильно разрешено отходить от seed-треков.
  • energy_curve: желаемая энергетическая траектория списка.
  • classifier_targets: какие promoted classifier scores считать желательными.
  • exclude_reviewed: исключать ли уже просмотренные кандидаты.

Поля выхода:

  • ранжированные track rows;
  • причина выбора для каждого трека;
  • similarity и diversity scores;
  • релевантные classifier scores;
  • опциональные данные порядка сета, например energy step или bridge target.

Пояснение: API лучше держать генеративным, но проверяемым. Он не должен просто возвращать список ID. Ему стоит возвращать достаточно контекста, чтобы UI мог показать пользователю, как был собран результат.

Возможные расширения позже ​

После первой версии можно добавить:

  • отправку выбранных треков в Rhythm Lab;
  • отметку кандидатов как reviewed;
  • экспорт M3U/XLSX списков;
  • сохранение сессий генерации;
  • итеративный active-learning workflow, где пользователь слушает кандидатов, размечает удачные и постепенно улучшает promoted classifiers.

Пояснение: это превращает идею из одноразового генератора списков в рабочий цикл. Сначала система предлагает кандидатов, потом пользователь проверяет их, затем эти решения становятся данными для следующих classifier-профилей и будущих подборок.
