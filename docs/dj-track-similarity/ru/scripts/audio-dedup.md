# Скрипт отчёта и очистки дублей аудио

По возможности запускайте этот скрипт через проектное Python-окружение:

```powershell
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --help
```

Вспомогательный инструмент для дублей аудио с двумя режимами. По умолчанию он
работает только как отчёт: читает существующую SQLite-базу
`dj-track-similarity`, сравнивает треки внутри выбранного сохранённого корня
путей и записывает отчёты JSON, оформленную книгу XLSX и текстовый лог. Отчёт
также проверяет базу Rhythm Lab по умолчанию
`tools\rhythm-lab\data\rhythm_lab.sqlite` и перечисляет строки лаборатории,
которые были бы удалены для безопасных кандидатов. JSON хранит компактный блок
`rhythm_lab.summary` и подробные затронутые строки; текстовый лог пишет ту же
сводку строкой `rhythm_lab_summary=...`.

С `--apply` добавляется подтверждённый проход очистки: после записи отчётов
скрипт удаляет безопасные файлы-дубли и их строки в основной базе данных. Если
база Rhythm Lab существует по пути по умолчанию, apply-режим также удаляет
соответствующие строки лаборатории для удалённых `source_track_id`.

Используйте режим отчёта по умолчанию, когда нужны доказательства возможных
дублей перед очисткой библиотеки. Он намеренно консервативен: отчёты всегда
создаются первыми. Разрушительная очистка запускается только с `--apply` и
интерактивным подтверждением, и удаляет только кандидатов, которых отчёт
пометил как безопасных.

Использование:

```text
python scripts\audio_dedup\audio_dedup.py --root ROOT [OPTIONS]
```

Параметры:

- `--db DB`: SQLite-база проекта. По умолчанию `C:\db\abstracted.sqlite`.
- `--root ROOT`: обязательный сохранённый корень путей, ограничивающий треки
  для сравнения.
- `--path-contains TEXT`: дополнительный фильтр пути без учёта регистра. Можно
  указывать несколько раз.
- `--preset safe|balanced|aggressive`: пресет оценки. По умолчанию `safe`.
- `--min-score SCORE`: переопределить порог включения дубля из пресета.
- `--limit-groups N`: записать не более N групп дублей.
- `--out-dir DIR`: каталог отчётов. По умолчанию
  `scripts\audio_dedup\reports`.
- `--apply`: после записи отчётов запросить точную фразу подтверждения и
  удалить только кандидатов, помеченных как `DELETE CANDIDATE` /
  `true_candidate`. Строки треков удаляются из SQLite только после успешного
  удаления их аудиофайлов.

## Пресеты

Каждый пресет задаёт порог попадания в отчёт, более строгий порог прямого
совпадения с keeper-треком для автоматического кандидата на удаление и
строгость совпадения длительности. `--min-score` переопределяет только порог
включения в отчёт; параметры безопасного удаления и длительности остаются как
в таблице.

| Пресет | Мин. оценка | Оценка safe delete | Допуск длительности | Когда использовать |
| --- | --- | --- | --- | --- |
| `safe` | `0.965` | `0.980` | ~2 с / 1% ratio | Консервативное обслуживание с минимальным числом ложных срабатываний. |
| `balanced` | `0.950` | `0.970` | ~5 с / 2.5% ratio | Более широкий поиск, который всё ещё может помечать сильных кандидатов на удаление. |
| `aggressive` | `0.925` | `0.965` | ~15 с / 8% ratio | Самый широкий поиск; ожидайте больше ручной проверки, но очень сильные прямые совпадения всё ещё могут стать delete candidates. |

## Оценка

Оценка пары объединяет доступные сигналы в одно значение `0..1`,
нормализованное по весу сигналов, которые есть у обоих треков:

| Сигнал | Вес |
| --- | --- |
| Сходство MERT embedding | `0.43` |
| Сходство MAEST embedding | `0.32` |
| Сходство признаков SONARA | `0.14` |
| Сходство CLAP embedding | `0.04` |
| Близость длительности | `0.05` |

Каждая группа получает уровень уверенности `high`, `medium` или `review` по
оценке и блокирующим причинам. BPM/key из тегов показываются только как
метаданные трека и никогда не используются для оценки дублей.

Примеры:

```powershell
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --db .\data\library.sqlite --root D:\Music
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --db .\data\library.sqlite --root D:\Music --preset balanced --path-contains mastered
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --db .\data\library.sqlite --root D:\Music --preset safe --apply
```

Файлы вывода называются `audio_dedup_report_<timestamp>.json`, `.xlsx` и
`.log`. Каталог отчётов по умолчанию игнорируется git.

После каждого запуска CLI печатает короткую сводку: число групп дублей, число
безопасных кандидатов в режиме отчёта и сведения о базе Rhythm Lab/затронутых
строках.

Книга XLSX является основным артефактом для ручной проверки. Она включает:

- `Summary`: путь к базе данных, выбранный корень, пресет, порог оценки, общее
  число треков в базе, число треков внутри выбранного корня/фильтра, количество
  групп и кандидатов, разбивку уверенности (`high`/`medium`/`review`) и
  счётчики покрытия embeddings MERT/MAEST/CLAP.
- `Groups`: по одной строке на группу дублей, с предлагаемым `KEEP`-треком и
  причинами, почему он предпочтительнее остальных файлов.
- `Candidates`: по одной строке на кандидата, с действием `DELETE CANDIDATE`
  или `REVIEW MANUALLY`, путём keeper-трека, прямой оценкой, доказательствами
  сходства и блокерами проверки.
- `Pair Evidence`: подробные попарные доказательства MERT, MAEST, SONARA, CLAP
  и длительности, использованные при группировке. BPM/key из тегов показаны
  только как метаданные и не участвуют в оценке дублей; BPM из SONARA остаётся
  частью сигнала сходства SONARA, когда доступен.
- `Rhythm Lab`: строки в базе Rhythm Lab по умолчанию, которые apply-режим
  удалил бы для кандидатов, помеченных безопасными. Если базы нет или у
  безопасных кандидатов нет строк лаборатории, лист содержит только заголовки.

Проверяйте каждого кандидата вручную; отчёт содержит предлагаемые keeper-треки
и доказательства для удаления кандидатов.

Apply-режим:

1. Скрипт сначала записывает отчёты JSON/XLSX/лог.
2. Он считает только безопасных кандидатов на удаление.
3. Печатает предупреждение о разрушительном действии и требует ввести точно
   `APPLY DELETE`.
4. Удаляет только файлы, которые всё ещё находятся внутри выбранного `--root`.
5. Удаляет строки SQLite только для треков, чьи файлы были успешно удалены.
   Связанные строки в таблицах со столбцом `track_id` удаляются перед строкой
   `tracks`.
6. Открывает `tools\rhythm-lab\data\rhythm_lab.sqlite`, если она существует, и
   удаляет строки, где `source_track_id` совпадает с успешно удалёнными track
   IDs. Это очищает ручные labels и predictions, не затрагивая classifier
   profiles и training checkpoints.
7. Перезаписывает JSON и лог с `mode = apply` и блоком `apply_result`, где
   перечислены удалённые track IDs, удалённые пути, пропущенные или неудачные
   удаления и число удалённых строк Rhythm Lab. Лог также пишет блок
   `deleted_files:` со строкой `deleted_file=...` для каждого файла, успешно
   удалённого с диска. Существующий блок `rhythm_lab.summary` в JSON остаётся
   pre-apply сводкой воздействия, показанной в книге.

Если вы отклоните подтверждение, отчёты останутся на диске и ничего не будет
удалено. Не используйте `--apply`, пока не проверите сгенерированную книгу.
Автоматические тесты и рутинная проверка не должны запускать скрипт с
`--apply`.

Для обычного обслуживания библиотеки начинайте с пресета `safe`. Используйте
`balanced` или `aggressive` только если готовы проверять больше ложных
срабатываний.
