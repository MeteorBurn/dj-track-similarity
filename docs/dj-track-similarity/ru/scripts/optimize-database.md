# Скрипт оптимизации базы данных

По возможности запускайте этот скрипт в проектном окружении Python:

```powershell
.\.venv\Scripts\python.exe scripts\optimize_database.py --help
```

Оптимизирует поддерживаемую проектную базу данных SQLite. Скрипт распознаёт
основную библиотечную базу и базу меток Rhythm Lab по маркерным таблицам,
создаёт резервную копию, выполняет `VACUUM`, `ANALYZE` и проверяет
целостность. Он не мигрирует, не восстанавливает и не адаптирует схемы баз
данных.

Используйте этот скрипт после крупного сканирования, сброса, очистки,
переноса, анализа, разметки или пересчёта предсказаний, если файл SQLite вырос
или вам нужна свежая проверка целостности. Он не входит в обычное повседневное
использование и не должен применяться для устранения проблем со схемой или
данными.

Использование:

```text
python scripts\optimize_database.py --db DB
```

Пример:

```powershell
python scripts\optimize_database.py --db .\data\library.sqlite
python scripts\optimize_database.py --db tools\rhythm-lab\data\rhythm_lab.sqlite
```

Этот скрипт пишет в базу данных и создаёт резервную копию рядом с ней. Если
база данных не распознана как основная библиотечная база или база Rhythm Lab,
скрипт выводит ошибку и останавливается до создания резервной копии или
изменения базы данных.

Существует только один режим работы и нет dry-run: каждый успешный запуск
проверяет базовые маркеры поддержки и целостность, создаёт резервную копию и
оптимизирует файл на месте.

## Как это работает

1. Требует, чтобы файл `--db` существовал.
2. Выполняет `PRAGMA integrity_check`; прерывается, если результат не `ok`.
3. Определяет поддерживаемый тип базы по маркерным таблицам:
   - основная библиотека: `tracks` и `embeddings`
   - Rhythm Lab: `classifier_profiles`, `classifier_labels`,
     `classifier_predictions` и `classifier_training_checkpoints`
   Неизвестные SQLite-файлы прерывают работу до создания резервной копии.
4. Создаёт онлайн-резервную копию SQLite с именем `<db-name>.bak-<YYYYMMDD-HHMMSS>`
   рядом с базой данных (если такое имя уже существует, добавляется числовой
   суффикс).
5. Включает журналирование WAL, затем выполняет `VACUUM`, `ANALYZE`,
   `PRAGMA optimize` и усекающую контрольную точку WAL.
6. Снова выполняет `PRAGMA integrity_check`; прерывается, если результат не `ok`.

## Вывод

В случае успеха скрипт выводит короткую сводку в стандартный вывод:

```text
database=<resolved database path>
database_kind=<library|rhythm_lab>
backup=<backup file path>
integrity_before=ok
integrity_after=ok
size_before=<bytes>
size_after=<bytes>
```

Единственный файл, который скрипт пишет помимо оптимизированной базы данных, —
это резервная копия `<db-name>.bak-<timestamp>`. Никаких отчётов в формате JSON,
XLSX или лога нет.

> Внимание: этот скрипт предназначен только для распознанных проектных баз
> данных. Он отказывается работать с посторонними SQLite-файлами, не мигрирует
> схемы и не запрашивает подтверждения перед оптимизацией.

Закройте работающее приложение или Rhythm Lab перед оптимизацией той же базы
данных, чтобы резервное копирование, `VACUUM` и проверка целостности
выполнялись на спокойном файле.
