# Скрипт отчёта и очистки дублей аудио

По возможности запускайте этот скрипт через проектное Python-окружение:

```powershell
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --help
```

Вспомогательный инструмент для поиска дублей аудио с двумя режимами. По
умолчанию он работает только на чтение: читает существующую базу данных SQLite
`dj-track-similarity`, сравнивает треки внутри выбранного корня сохранённых
путей и записывает отчёты в форматах JSON, оформленном XLSX и текстовом логе. С
флагом `--apply` добавляется подтверждённый проход очистки, который удаляет
безопасные файлы-дубли и их строки в базе данных после записи отчётов.

Используйте режим отчёта по умолчанию, когда нужны доказательства возможных
дублей аудио перед очисткой библиотеки. Он намеренно консервативен: сначала
всегда формирует отчёты. Разрушительная очистка запускается только с флагом
`--apply` и интерактивным запросом подтверждения, и удаляет только тех
кандидатов, которых отчёт пометил как безопасные.

Использование:

```text
python scripts\audio_dedup\audio_dedup.py --root ROOT [OPTIONS]
```

Параметры:

- `--db DB`: база данных проекта SQLite. По умолчанию `C:\db\abstracted.sqlite`.
- `--root ROOT`: обязательный корень сохранённых путей, ограничивающий круг
  треков-кандидатов.
- `--path-contains TEXT`: дополнительный фильтр по пути без учёта регистра.
  Можно указывать несколько раз.
- `--preset safe|balanced|aggressive`: пресет оценки. По умолчанию `safe`.
- `--min-score SCORE`: переопределяет порог дубля, заданный пресетом.
- `--limit-groups N`: записать не более N групп дублей.
- `--out-dir DIR`: каталог для отчётов. По умолчанию
  `scripts\audio_dedup\reports`.
- `--apply`: после записи отчётов запрашивает точную фразу подтверждения и
  удаляет только кандидатов, помеченных как `DELETE CANDIDATE` /
  `true_candidate`. Треки удаляются из SQLite только после успешного удаления
  их аудиофайлов.

## Пресеты

Каждый пресет задаёт порог оценки дубля по умолчанию и строгость совпадения по
длительности. Флаг `--min-score` переопределяет только порог оценки; остальные
параметры пресета остаются прежними.

| Пресет | Мин. оценка | Допуск по длительности | Когда использовать |
| --- | --- | --- | --- |
| `safe` | `0.965` | ~2 с / 1% соотношения | Консервативное обслуживание с минимумом ложных срабатываний. |
| `balanced` | `0.925` | ~5 с / 2.5% соотношения | Более широкий охват, когда вы готовы проверять больше кандидатов. |
| `aggressive` | `0.875` | ~15 с / 8% соотношения | Самый широкий поиск совпадений; ожидайте больше ручной проверки. |

## Оценка

Оценка пары объединяет доступные сигналы в одно значение в диапазоне `0..1`,
нормализованное по весу сигналов, существующих для обоих треков:

| Сигнал | Вес |
| --- | --- |
| Сходство embedding MERT | `0.43` |
| Сходство embedding MAEST | `0.32` |
| Сходство признаков SONARA | `0.14` |
| Сходство embedding CLAP | `0.04` |
| Близость длительности | `0.05` |

Каждой группе присваивается уровень уверенности `high`, `medium` или `review` на
основе её оценки и любых блокирующих причин. Значения BPM/тональности из тегов
показываются только как метаданные трека и никогда не используются для оценки
дублей.

Примеры:

```powershell
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --db .\data\library.sqlite --root D:\Music
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --db .\data\library.sqlite --root D:\Music --preset balanced --path-contains mastered
.\.venv\Scripts\python.exe scripts\audio_dedup\audio_dedup.py --db .\data\library.sqlite --root D:\Music --preset safe --apply
```

Файлы вывода именуются `audio_dedup_report_<timestamp>.json`, `.xlsx` и `.log`.
Каталог отчётов по умолчанию игнорируется git.

Книга XLSX — основной артефакт для ручной проверки человеком. Она включает:

- `Summary`: путь к базе данных, выбранный корень, пресет, порог оценки, общее
  число треков в базе данных, число треков внутри выбранного корня/фильтра,
  количество групп дублей и кандидатов, разбивку по уверенности
  (`high`/`medium`/`review`) и счётчики покрытия embedding MERT/MAEST/CLAP.
- `Groups`: по одной строке на группу дублей, с предлагаемым треком `KEEP` и
  причинами, по которым он опережает остальные файлы.
- `Candidates`: по одной строке на кандидата-дубль, с пометкой `DELETE CANDIDATE`
  или `REVIEW MANUALLY`, путём оставляемого трека, прямой оценкой,
  доказательствами сходства и блокирующими факторами проверки.
- `Pair Evidence`: подробные попарные доказательства по MERT, MAEST, SONARA, CLAP
  и длительности, используемые на шаге группировки. Значения BPM/тональности из
  тегов показываются только как метаданные трека и не используются для оценки
  дублей; BPM из SONARA остаётся частью сигнала сходства SONARA, когда доступен.

Проверяйте каждого кандидата вручную; отчёт содержит предлагаемые оставляемые
треки и доказательства для удаления кандидатов.

Режим применения:

1. Сначала скрипт записывает отчёты JSON/XLSX/лог.
2. Затем подсчитывает только безопасных кандидатов на удаление.
3. Выводит предупреждение о разрушительном действии и требует ввести в точности
   `APPLY DELETE`.
4. Удаляет только файлы, всё ещё находящиеся внутри выбранного `--root`.
5. Удаляет строки SQLite только для треков, чьи файлы были успешно удалены.
   Связанные строки в таблицах со столбцом `track_id` удаляются перед строкой в
   `tracks`.
6. Перезаписывает отчёты JSON и лог с `mode`, равным `apply`, и блоком
   `apply_result`, перечисляющим идентификаторы удалённых треков, удалённые пути
   и любые пропущенные или неудавшиеся удаления.

Если вы отклоните подтверждение, отчёты останутся на диске, и ничего не будет
удалено. Не используйте `--apply`, пока не проверите сформированную книгу XLSX.
Автоматические тесты и рутинная проверка не должны вызывать скрипт с `--apply`.

Начинайте с пресета `safe` для обычного обслуживания библиотеки. Используйте
`balanced` или `aggressive` только тогда, когда готовы проверять больше ложных
срабатываний.
